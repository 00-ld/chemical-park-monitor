# 化工园区危险气体扩散与溯源系统项目总体要求

> 本文档作为项目需求、架构设计、开发实现、测试验证、部署运维的总体约束文件。后续代码、数据库、接口、算法、页面和文档均应遵循本文档；若实际实现与本文档冲突，应先更新本文档并记录变更原因。

## 1. 项目定位

本项目围绕化工园区危险气体安全管控，建设集“危险气体扩散模拟、泄漏源溯源、传感器监测、巡检小车动态采样、逃生路径规划、二维/三维数字孪生可视化、安全联动决策”于一体的综合平台。

系统应支持在二维或三维数字孪生底座上选取泄漏点位，配置风速、风向、气体类型、泄漏速率、环境参数、气体物理参数等条件，模拟真实环境下的危险气体扩散过程；同时结合固定传感器与阿克曼巡检小车采集的浓度数据，进行泄漏源反演溯源，并根据气体扩散浓度场动态生成最优逃生路径。

## 2. 核心目标

1. 建立经过真实性验证的危险气体扩散模型，支持化工园区典型泄漏场景模拟。
2. 建立基于多点位浓度数据的泄漏源溯源模型，输出泄漏点位的估计位置、置信度和误差范围。
3. 建立依据国家标准或行业规范的传感器监控点位布局方案，支持固定点位与动态巡检点位协同。
4. 建立实时或准实时的逃生路径规划能力，路径应随危险气体浓度场变化动态更新。
5. 建立二维与三维联动的数字孪生展示能力，二维优先承载监控点位布局、浓度热力图和路径规划，三维作为最终核心展示目标。
6. 建立规范化 API、数据库、算法、日志、安全和部署约束，使项目具备持续维护和服务器部署能力。

## 3. 技术栈约束

### 3.1 前端

- 使用 Vue 3 作为前端主体框架。
- 使用 ECharts 承载统计图表、趋势曲线、浓度分布图等可视化内容。
- 使用 Canvas 承载二维园区平面、浓度热力图、监控点位、路径规划等高频绘制内容。
- 前端应模块化组织页面、组件、API 请求、状态管理和工具方法。
- 前端所有接口调用必须通过统一请求封装，不允许在页面组件中散落硬编码接口地址。

### 3.2 后端

- 后端主体使用 Python。
- 推荐使用 FastAPI 构建规范化接口服务。
- 后端必须遵循统一接口协议，所有业务接口、算法接口、设备接口和管理接口均应返回统一响应结构。
- 后端应分层清晰，至少包含 API 层、服务层、业务逻辑层、工具层、算法调用层和数据访问层。

### 3.3 三维数字孪生

- 三维底座可使用 SuperMap 或 Three.js。
- 已在 SuperMap iPortal 中建设的数字大屏应作为三维数字孪生的重要展示入口，后端需要为 iPortal 数字大屏提供稳定、统一、标准化的 JSON 数据接口。
- 三维展示应支持园区建筑、道路、设备、监控点位、巡检小车、泄漏点、浓度云图或体渲染效果。
- 三维坐标与二维坐标必须建立统一映射关系，所有点位数据必须具备可追溯坐标来源。

### 3.4 数据库

- 使用 MySQL 作为主要业务数据库。
- 项目根目录必须包含 `db/` 文件夹，用于存放数据库初始化脚本、迁移脚本、种子数据、测试数据说明和 ER 图。
- 数据库字符集必须统一使用 `utf8mb4`，排序规则建议使用 `utf8mb4_unicode_ci` 或项目统一约定的兼容规则。

### 3.5 AI 与开发辅助技术

- 可使用 OpenAI、Claude Code、DeepSeek、Mimo 等 AI 技术辅助开发、文档生成、代码审查、算法实验和测试数据分析。
- 后续可接入千问大模型，用于逃生规范建议、事故处理建议、应急预案问答、异常数据解释和事故报告辅助生成。
- AI 生成的代码、算法结论、测试报告和文档必须经过人工复核。
- 涉及安全决策、人员疏散、泄漏研判的结果必须保持客观表达，不得夸大模型能力。
- 大模型输出只能作为辅助建议，不得直接替代扩散模型、溯源模型、路径规划算法和现场应急负责人决策。

### 3.6 硬件

- 固定气体传感器作为静态监控点位。
- 阿克曼巡检小车作为动态监控点位。
- 小车应支持搭载气体传感器和摄像头。
- 小车摄像头上传的图片或视频可通过 YOLO11m 算法识别人员位置。

## 4. 监测气体范围

系统首期必须支持以下四类气体浓度数据：

| 气体 | 标识 | 说明 |
| --- | --- | --- |
| 一氧化碳 | CO | 有毒气体 |
| 氧气 | O2 | 环境氧含量，需关注缺氧或富氧风险 |
| 氨气 | NH3 | 有毒、有刺激性气体 |
| 甲烷 | CH4 | 易燃易爆气体 |

所有气体数据必须包含单位、采样时间、采样点位、数据来源、设备编号和质量标记。

## 5. 总体架构分层

项目至少包含以下逻辑层次，各层职责必须清晰，禁止跨层直接耦合。

```text
前端展示层
  -> API 接口层
  -> 业务逻辑层
  -> 服务层
  -> 算法实现层
  -> tools 工具层
  -> 数据访问层
  -> MySQL / 文件 / 模型 / 日志
  -> 三维可视化层
```

### 5.1 项目根目录规范化子文件夹

项目根目录下必须建立清晰、稳定、可维护的子文件夹结构。所有新增代码、数据、文档、模型、脚本和部署配置都应放入对应目录，禁止长期堆放在项目根目录。

建议根目录结构如下：

```text
.
  frontend/        # 前端工程，Vue 3、ECharts、Canvas、页面和组件
  backend/         # Python 后端 API 服务，统一接口、业务服务、权限认证
  algorithm/       # 算法实现，扩散、溯源、路径规划、YOLO 推理封装
  twin/            # 三维数字孪生，SuperMap、Three.js、iPortal 大屏接入
  db/              # MySQL 初始化脚本、迁移脚本、种子数据、ER 图
  datasets/        # 权威数据集、原始数据、清洗后数据、数据来源说明
  models/          # 训练模型、推理模型、模型版本说明，不提交超大敏感模型
  docs/            # 项目文档、接口文档、算法验证报告、部署文档
  tests/           # 单元测试、接口测试、算法测试、端到端测试
  tools/           # 通用工具脚本，不绑定具体业务流程
  scripts/         # 开发、构建、数据处理、部署辅助脚本
  docker/          # Dockerfile、容器配置、服务镜像构建文件
  deploy/          # 服务器部署、Nginx、HTTPS、域名、运维配置
  config/          # 配置模板，不存放真实密钥
  uploads/         # 本地上传文件临时目录，生产环境应挂载到独立存储
  logs/            # 本地运行日志目录，禁止提交真实生产日志
  assets/          # 图片、图标、地图静态资源、三维模型轻量资源
  .github/         # CI、代码检查、自动化流程
```

强制约束：

- 根目录只允许保留 README、项目配置文件和少量入口文件。
- 业务代码不得直接散落在根目录。
- 真实密钥、生产数据库备份、生产日志、未脱敏数据集不得提交到仓库。
- 若当前仓库存在历史目录名，例如 `frontend/`、`backend/`、`algorithm/`，应在文档中明确其与规范目录的映射关系，后续逐步迁移或保持单一命名规范。
- 每个一级目录应至少包含 `README.md` 或等价说明，解释该目录用途、维护规则和禁止事项。

## 6. 业务逻辑层约束

业务逻辑层负责组织系统核心业务流程，不直接处理底层数据库细节，不直接实现复杂算法。

必须覆盖以下业务：

1. 泄漏场景创建：选择泄漏点、气体类型、泄漏参数、环境参数。
2. 扩散模拟任务管理：创建任务、启动任务、暂停任务、查询进度、查看结果。
3. 浓度数据管理：接收固定传感器、小车动态采样和仿真模型输出数据。
4. 异常识别：根据阈值、趋势、空间分布判断异常浓度数据。
5. 泄漏源溯源：基于异常点位浓度数据调用溯源模型。
6. 安全决策联动：生成异常日志、风险等级、建议处置动作和联动通知。
7. 逃生路径规划：基于人员位置、道路拓扑、气体浓度场动态计算最优路径。
8. 小车任务管理：管理小车状态、位置、采样任务、摄像头识别结果和异常上报。

## 7. 服务层约束

服务层负责封装可复用业务能力，对 API 层提供稳定服务。

建议拆分为以下服务：

| 服务 | 职责 |
| --- | --- |
| SensorService | 传感器注册、状态、点位、采样数据处理 |
| GasDataService | 四类气体浓度数据入库、查询、统计、异常筛选 |
| DiffusionService | 扩散模型任务调度、参数校验、结果管理 |
| SourceInversionService | 溯源模型调用、结果解释、误差评估 |
| EvacuationService | D* Lite 路径规划、风险区域避让、路径更新 |
| VehicleService | 阿克曼小车管理、动态采样、视频/图片上传记录 |
| VisionService | YOLO11m 图片/视频识别、人员位置输出 |
| TwinService | 二维/三维坐标转换、园区对象管理、可视化数据聚合 |
| AlarmService | 异常日志、告警等级、联动处置记录 |
| UserService | 人员、角色、权限、登录认证 |

服务层不得直接返回数据库实体给前端，必须通过 DTO 或统一响应模型输出。

## 8. tools 层约束

tools 层用于存放无业务状态的通用工具，必须保持可复用、可测试、低耦合。

建议包含：

- 坐标转换工具：二维坐标、三维坐标、GIS 坐标转换。
- 单位换算工具：ppm、mg/m3、体积分数等单位转换。
- 时间处理工具：时区、时间戳、采样间隔、仿真步长处理。
- 数据校验工具：气体参数、环境参数、传感器数据合法性校验。
- 文件处理工具：上传文件、模型文件、数据集文件、日志文件处理。
- 加密与脱敏工具：密码哈希、敏感字段脱敏、token 处理。
- API 响应工具：统一响应、错误码、分页结构。
- 日志工具：业务日志、异常日志、算法运行日志、审计日志。

tools 层不得依赖页面、具体业务流程或数据库表结构。

## 9. 算法实现层约束

算法实现层是项目核心，应独立组织、可测试、可复现实验结果。

### 9.1 扩散模型

扩散模型必须支持：

- 泄漏点坐标。
- 气体类型。
- 泄漏速率或源强。
- 风速、风向。
- 温度、湿度、稳定度等环境参数。
- 地形或障碍物影响的扩展接口。
- 输出二维浓度场，预留三维浓度场能力。

扩散结果必须包含：

- 时间步。
- 空间坐标。
- 浓度值。
- 气体类型。
- 模型参数快照。
- 结果可信说明。

### 9.2 溯源模型

溯源模型必须支持基于多个监控点位浓度数据进行泄漏点反演。

输出至少包含：

- 估计泄漏点坐标。
- 估计源强。
- 置信度或可信区间。
- 与真实泄漏点的误差。
- 参与计算的传感器点位列表。
- 模型版本和参数快照。

### 9.3 逃生路径规划

逃生路径规划使用 D* Lite 算法作为核心动态路径规划算法。

路径规划必须考虑：

- 人员当前位置。
- 安全出口位置。
- 园区道路或可通行区域拓扑。
- 当前及预测浓度场。
- 高风险区域避让。
- 气体扩散导致的动态重规划。

当道路或区域浓度达到禁止通行阈值时，路径规划必须剔除该区域，不得仅通过降低权重继续通行。

### 9.4 YOLO11m 识别

小车管理页面必须支持上传小车摄像头拍摄的图片或视频，并调用 YOLO11m 算法识别人员位置。

识别结果应包含：

- 人员框坐标。
- 置信度。
- 图片或视频帧编号。
- 小车编号。
- 拍摄时间。
- 识别模型版本。

### 9.5 算法真实性验证

每个核心算法至少需要完成 5 次测试验证真实性。

验证要求：

1. 测试数据必须来自真实数据集、权威网站、权威平台、公开实验数据或可说明来源的现场采样数据。
2. 数据集必须下载并归档，不能只引用网页描述。
3. 每次验证必须记录数据来源、下载时间、数据版本、预处理方法、测试参数、指标结果和结论。
4. 验证结论必须保持客观，不得只写“效果良好”，必须说明误差、适用边界和失败情况。
5. 若使用仿真数据辅助训练，必须明确标注为仿真数据，不得当作真实数据。

建议建立如下文件：

```text
datasets/
  README.md
  raw/
  processed/
  metadata/
algorithm_tests/
  diffusion/
  source_inversion/
  evacuation/
  vision/
docs/
  algorithm-verification-report.md
```

## 10. 三维可视化层约束

三维可视化层负责展示园区数字孪生底座和安全态势。

必须支持：

- 园区建筑、道路、装置、罐区、管线或重点风险区域展示。
- 固定传感器点位展示。
- 阿克曼巡检小车位置展示。
- 泄漏点展示。
- 气体浓度场展示。
- 告警区域展示。
- 逃生路径展示。
- 二维与三维视角联动。
- SuperMap iPortal 数字大屏数据接入。

三维展示应优先保证信息准确和交互清晰，不得为了视觉效果牺牲安全信息可读性。

### 10.1 SuperMap iPortal 数字大屏接入

已建设的 SuperMap iPortal 数字大屏应纳入系统总体架构，作为园区态势展示和三维成果发布的重要入口。

后端必须为 iPortal 数字大屏提供标准化 JSON 数据接口，至少覆盖：

- 园区总体安全状态。
- 传感器实时点位数据。
- 四类气体浓度实时值和历史趋势。
- 异常告警列表。
- 泄漏点估计结果。
- 扩散浓度场概要数据。
- 小车实时位置和状态。
- 人员位置或区域统计。
- 逃生路径结果。
- 应急处置建议和联动状态。

iPortal 接口约束：

- 返回结构必须遵循本项目统一 JSON 响应协议。
- 坐标字段必须明确坐标系、二维坐标、三维坐标和来源。
- 时间字段必须使用 ISO 8601 格式并包含时区。
- 大屏高频刷新接口应控制返回体大小，必要时提供摘要接口和详情接口。
- 三维场景对象应包含稳定唯一 ID，便于前端或 iPortal 做增量更新。

## 11. API 接口层约束

### 11.1 统一接口前缀

所有后端接口必须统一使用版本化路径：

```text
/api/v1/...
```

算法服务接口建议使用：

```text
/api/v1/algorithm/...
```

设备接口建议使用：

```text
/api/v1/device/...
```

### 11.2 统一响应协议

所有接口必须返回统一 JSON 结构：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "data": {},
  "meta": {
    "page": null,
    "page_size": null,
    "total": null
  },
  "request_id": "uuid",
  "timestamp": "2026-06-12T09:00:00+08:00"
}
```

失败响应示例：

```json
{
  "success": false,
  "code": "VALIDATION_ERROR",
  "message": "invalid gas type",
  "data": null,
  "meta": null,
  "request_id": "uuid",
  "timestamp": "2026-06-12T09:00:00+08:00"
}
```

字段规范：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| success | boolean | 是 | 请求是否成功 |
| code | string | 是 | 稳定错误码或成功码，禁止直接返回临时中文字符串作为 code |
| message | string | 是 | 给前端或调用方展示的简短说明 |
| data | object/array/null | 是 | 业务数据主体，失败时通常为 null |
| meta | object/null | 是 | 分页、统计、单位、坐标系、算法版本等附加信息 |
| request_id | string | 是 | 请求唯一标识，用于日志追踪 |
| timestamp | string | 是 | 服务端响应时间，必须包含时区 |

分页接口 `meta` 必须包含：

```json
{
  "page": 1,
  "page_size": 20,
  "total": 100
}
```

算法接口 `meta` 建议包含：

```json
{
  "algorithm": "source_inversion",
  "model_version": "v1.0.0",
  "dataset_version": "dataset-2026-06",
  "unit": "ppm",
  "coordinate_system": "park_local_3d"
}
```

坐标类接口 `data` 或 `meta` 必须明确：

```json
{
  "coordinate_system": "park_local_3d",
  "x": 120.5,
  "y": 36.2,
  "z": 8.0,
  "source": "supermap_iportal"
}
```

后端、前端、iPortal 数字大屏、算法服务、小车设备服务均必须遵循同一套 JSON 返回规范。

### 11.3 错误码规范

错误码必须稳定、可检索、可文档化。

| 错误码 | 含义 |
| --- | --- |
| OK | 请求成功 |
| VALIDATION_ERROR | 参数校验失败 |
| AUTH_REQUIRED | 未登录或 token 缺失 |
| PERMISSION_DENIED | 无权限 |
| RESOURCE_NOT_FOUND | 资源不存在 |
| DEVICE_OFFLINE | 设备离线 |
| SENSOR_DATA_INVALID | 传感器数据异常 |

## 18. 核心业务流程

### 18.1 泄漏扩散与溯源流程

```text
创建泄漏场景
  -> 设置环境参数和气体参数
  -> 扩散模型生成浓度场
  -> 固定传感器/小车点位获得浓度数据
  -> 异常浓度识别
  -> 溯源模型反演泄漏点
  -> 输出泄漏点估计结果
  -> 三维/二维可视化展示
  -> 生成异常日志和联动建议
```

### 18.2 小车异常联动流程

```text
小车巡检
  -> 摄像头上传图片/视频
  -> YOLO11m 识别人员位置
  -> 传感器检测气体浓度
  -> 发现异常浓度
  -> 自动记录异常日志
  -> 跳转扩散与溯源页面
  -> 汇总全部监控点位浓度数据
  -> 溯源泄漏点
  -> 触发安全决策和路径规划
```

### 18.3 逃生路径流程

```text
获取人员位置
  -> 获取安全出口
  -> 获取道路/可通行区域拓扑
  -> 获取当前浓度场
  -> 剔除高危区域
  -> D* Lite 计算最优路径
  -> 气体扩散变化后动态重规划
  -> 二维优先展示，三维同步预留
```

### 18.4 千问大模型辅助决策流程

千问大模型后续可作为系统的辅助决策与知识问答模块接入，但必须基于扩散模型、溯源模型、路径规划模型、传感器数据和应急规范文档提供建议。

建议流程：

```text
获取当前事故上下文
  -> 汇总气体类型、浓度、泄漏估计点、人员位置、风速风向、告警等级
  -> 检索应急预案、国家标准、企业处置规范和历史案例
  -> 千问大模型生成逃生规范建议、事故处理建议和沟通话术
  -> 后端标注建议来源、可信边界和人工复核要求
  -> 前端展示为辅助建议，不作为强制指令
  -> 处置人员确认后形成联动记录
```

可接入场景：

- 逃生规范建议：根据气体类型、浓度、人员位置和安全出口，生成面向人员的疏散提醒。
- 事故处理建议：根据泄漏类型、风险等级和现场条件，生成处置步骤建议。
- 应急预案问答：基于本地预案文档、国家标准和企业制度回答问题。
- 异常数据解释：解释某段时间内传感器异常、浓度突变或小车采样异常的可能原因。
- 事故报告辅助生成：根据告警日志、传感器数据、溯源结果和处置记录生成初稿。
- 值班人员辅助研判：将复杂图表、浓度趋势、路径变化转换为更容易理解的文字说明。
- 培训与演练：根据历史事故场景生成演练问答、考核题和处置复盘材料。
- 接口与日志分析：辅助研发人员分析接口异常、模型调用失败和数据质量问题。

大模型接入限制：

- 不得直接控制小车、告警、门禁、广播或任何现场执行设备。
- 不得绕过扩散、溯源和路径规划算法直接给出“确定性”结论。
- 涉及安全处置的输出必须显示“辅助建议，需人工复核”。
- 所有大模型请求和响应必须记录 request_id、模型名称、模型版本、输入摘要和输出摘要。
- 不得向大模型发送未脱敏的用户密码、API Key、数据库密码或无关个人敏感信息。
- 大模型不可用时，系统核心监测、扩散、溯源、路径规划和告警功能必须仍可运行。
