# 主页模块规范

本文档约束 `frontend/src/views/home/` 主页模块的页面职责、数据来源、接口调用、视觉风格和提交规则。后续修改 `src/views/home/index.vue` 前，应先确认本规范是否需要同步更新。

## 1. 页面定位

主页是系统总体态势可视化入口，主要用于快速查看化工园区安全运行状态，不承担复杂管理、数据维护、算法参数配置或事故处置确认功能。

主页必须保持“总览、预警、趋势、入口”的定位：

- 展示园区总体安全态势。
- 展示关键指标和近期异常。
- 展示气体浓度、预警类型、设备状态等统计图表。
- 提供跳转到核心业务页面的快捷入口。
- 不在主页直接执行删除、审批、强制处置、设备控制等高风险操作。

## 2. 路由与文件位置

主页路由固定为：

```text
/home
```

主页文件位置固定为：

```text
frontend/src/views/home/index.vue
frontend/src/views/home/README.md
```

命名约束：

- 目录名使用小写英文 `home`。
- 主页入口组件使用 `index.vue`。
- 不得新增 `HomePage.vue`、`home_test.vue`、`首页.vue`、`new_home.vue` 等重复入口文件。
- 临时实验内容不得长期放在 `home/` 目录下，应移动到规范目录或删除。

## 3. 展示内容要求

主页至少应覆盖以下信息，数据为空时必须显示合理空状态：

| 模块 | 展示内容 | 说明 |
| --- | --- | --- |
| 核心指标卡片 | 企业数、设备数、待处理预警、入园人员、入园车辆、设备在线率 | 指标应来自后端或可说明来源的模拟数据 |
| 预警分布图 | 最近一段时间内 CO、O2、NH3、CH4 等预警类型分布 | 使用 ECharts 展示，禁止写死不可解释数据 |
| 浓度趋势图 | 园区气体浓度趋势 | 首期可使用 VOCs 或四类气体趋势，后续应统一为 CO、O2、NH3、CH4 |
| 快捷入口 | 智巡监测、人员管理、实时预警、实时监测、智慧地图 | 只负责跳转，不在主页承载管理表单 |
| 园区公告 | 安全通知、巡检通知、维护通知 | 后续应接入后端公告接口 |
| 更新时间 | 页面数据最后更新时间 | 刷新后应同步更新 |

主页展示的气体范围应与项目总体要求保持一致：

```text
CO, O2, NH3, CH4
```

如页面中保留 VOCs 趋势等历史展示，应在后续需求中逐步替换或补充为项目约定的四类气体数据。

## 4. 数据来源与接口规范

主页数据应优先通过后端统一接口获取，不得在组件中散落硬编码第三方接口地址。

接口调用约束：

- 所有业务接口应通过统一请求封装或 `src/api/` 模块调用。
- 不得在主页组件中写入真实 API Key、token、数据库密码或第三方密钥。
- 不得从浏览器端直连需要密钥的第三方服务。
- 本地开发地址、后端地址和算法服务地址必须通过 `.env.*` 中的公开配置或统一代理处理。
- 后端响应必须遵循项目统一 JSON 返回规范。

统一 JSON 返回格式：

```json
{
  "code": 200,
  "message": "成功",
  "data": {},
  "ok": true,
  "timestamp": 1781234567890,
  "requestId": "uuid"
}
```

主页接口建议拆分：

| 接口用途 | 建议路径 | 返回内容 |
| --- | --- | --- |
| 总览指标 | `/api/dashboard/summary` | 企业数、设备数、人员数、车辆数、在线率、预警数 |
| 预警分布 | `/api/dashboard/alarm-distribution` | 气体类型、预警等级、数量 |
| 浓度趋势 | `/api/dashboard/gas-trend` | 时间序列、气体类型、浓度值、单位 |
| 最近异常 | `/api/dashboard/recent-alarms` | 异常编号、气体类型、浓度、位置、时间、状态 |
| 园区公告 | `/api/dashboard/notices` | 标题、内容摘要、发布时间、公告等级 |

如果接口暂未完成，可以使用可追踪的模拟数据，但必须满足：

- 模拟数据集中放在明确的数据模块或组件内清晰标注。
- 不得伪装成真实现场数据。
- 不得写成“实测”“真实采集”等误导性描述。
- 接口完成后应及时替换模拟数据。

## 5. 视觉与交互规范

主页视觉必须与系统现有深色工业监控风格一致：

- 保持深色背景、数据大屏感、清晰高对比指标。
- 不得改成营销落地页、宣传页或与系统风格无关的装饰页面。
- 卡片、图表、按钮的布局应适合值班人员快速扫描。
- 重要异常指标应有明确颜色区分，但不得过度闪烁或干扰阅读。
- 所有中文文案必须无乱码，编译后页面不得出现中文乱码。
- 页面在常见桌面分辨率下不得出现文字重叠、按钮溢出、图表挤压。

交互约束：

- 快捷按钮只负责路由跳转。
- 刷新按钮应只刷新对应图表或数据区域。
- 接口加载时应有 loading 状态。
- 接口失败时应有错误提示，并保留页面基本可用。
- 数据为空时应显示“暂无数据”或等价空状态，不得出现空白图表或报错堆栈。

## 6. ECharts 与渲染规范

主页图表使用 ECharts，必须注意性能和生命周期：

- 图表初始化应在 DOM 挂载后执行。
- 页面卸载时应释放 ECharts 实例，避免内存泄漏。
- 浏览器窗口变化时应调用 `resize()`。
- 图表数据更新应使用 `setOption()`，避免重复创建实例。
- 图表颜色、字体、背景应与系统主题保持一致。
- 不得在图表配置中混入无来源的大段临时数据。

如后续主页增加 Canvas 动态态势图，应满足：

- Canvas 只用于高频绘制或空间态势展示。
- 绘制逻辑应封装清晰，不与接口请求、路由跳转混在同一段复杂代码中。
- 动画循环必须可停止，组件卸载时必须清理。

## 7. 与其他页面的联动

主页允许跳转到以下核心页面：

| 入口 | 路由 | 用途 |
| --- | --- | --- |
| 智巡监测 | `/car/home` | 查看阿克曼巡检小车状态、位置、传感器数据 |
| 人员管理 | `/acl/employee` 或现有人员路由 | 查看和维护人员信息 |
| 实时预警 | `/thing/monitor_history` | 查看预警与异常记录 |
| 实时监测 | 按现有监控页面路由 | 查看实时监控数据 |
| 智慧地图 | `/smart-map` | 查看二维/三维园区、扩散与溯源、路径规划 |
| 数字园区 | `/screen` | 查看 SuperMap iPortal 数字大屏 |

联动要求：

- 主页只展示摘要和入口，详细操作应跳转到对应业务页面。
- 异常数量、最近异常和气体趋势应能与监控数据管理、扩散与溯源页面形成一致口径。
- 小车异常、传感器异常、扩散异常等信息不得在不同页面出现互相矛盾的状态。

## 8. 安全与客观表达

主页展示安全态势和算法相关结果时必须保持客观：

- 可以写“当前监测到异常”“模型估计风险升高”“建议前往详情页复核”。
- 不得写“绝对安全”“已完全确认泄漏点”“保证人员安全”等绝对化结论。
- 大模型建议后续接入主页时，只能作为辅助提示入口，不得直接替代现场决策。
- 不得展示后端异常堆栈、服务器路径、数据库信息或调试 token。

敏感信息约束：

- 不得提交真实 `.env`、API Key、token、MySQL 密码、用户密码。
- 不得在主页代码、注释、README 或构建产物中暴露密钥。
- 不得提交未脱敏真实用户数据、真实事故数据或生产日志。

## 9. 测试与验收

修改主页后至少执行：

```bash
cd frontend
npm run typecheck
```

涉及布局、图表或接口时建议同时执行：

```bash
cd frontend
npm run build:pro
```

验收标准：

- `/home` 可以正常访问。
- 主页刷新后不白屏。
- 图表可以正常初始化和刷新。
- 接口失败时页面有提示，不暴露调试信息。
- 中文显示正常，无乱码。
- 快捷入口跳转正确。
- `dist/`、截图、临时文件、缓存文件不得提交到 GitHub。

## 10. GitHub 提交约束

主页相关提交应保持聚焦：

- 修改主页视觉，只提交 `frontend/src/views/home/` 及必要样式/资源。
- 修改主页接口，只提交对应 API 封装、类型定义和必要文档。
- 修改路由，只提交路由配置和必要说明。
- 不得把无关页面、构建产物、临时图片、测试视频、个人笔记一起提交。
- 提交前必须检查 `git status`，确认没有多余文件进入 Git 跟踪。
